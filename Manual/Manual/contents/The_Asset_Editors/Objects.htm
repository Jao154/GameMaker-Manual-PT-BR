O Editor Object

Os objetos são o recurso que usamos para controlar aspectos de um jogo e para fazer coisas específicas. Na maioria das vezes eles têm um sprite associado a eles para que você os veja no jogo rooms, mas às vezes eles são usados como um controlador "nos bastidores" para fazer coisas relacionadas ao usuário ou para cronometrar, etc... Eles podem ter comportamentos e podem reagir a certos eventos, assim como uns aos outros, e a maioria das coisas que você vê em um jogo são baseadas em objects e suas interações. Agora, nós dizemos "baseado em" porque você não coloca objects diretamente no jogo rooms, mas coloca instâncias destes objects que são basicamente cópias (ou clones se preferir) do recurso object. Isto é uma coisa muito importante a ser lembrada como instâncias e objects não são a mesma coisa e cada uma tem seu próprio conjunto de funções que podem afetá-las.

Para esclarecer isto, pense em um jogo feito com o GameMaker. Todos os personagens, monstros, bolas, paredes, etc... são todos objects que você cria no Navegador de Ativos. Você então arrasta o objeto para um room no Editor de Sala para criar uma instância copiada do modelo básico object. A instância no room pode então ser alterada no próprio editor ou através de código, para que possa ser escalonada, ou colorida, ou ter outros detalhes modificados a partir do "base" object. Portanto, quando falamos de algo que afeta ou altera uma instância, queremos dizer que afeta uma cópia em particular de um object em uma sala. Somente aquela única instância está sendo afetada enquanto todas as outras não estão. Mas, quando falamos em afetar ou mudar um objeto, queremos dizer que estamos modificando o object no "Browser" Asset e qualquer coisa que fizermos a ele será refletida em todas as instâncias criadas a partir desse ponto também. Portanto, o object é o modelo para a instância, e a instância é o que colocamos em um room para fazer nosso jogo.

Para criar um object em seu jogo, você pode clicar com o botão direito do mouse no navegador Asset e clicar na opção Criar objeto. Isto abrirá a janela ilustrada no topo desta página com as seguintes seções:

Detalhes

Aqui você pode dar um nome ao seu object. Este nome nunca deve conter espaços nem caracteres não autorizados (somente caracteres alfanuméricos e a barra inferior "_" são permitidos) e deve ser curto, mas memorável, para que você possa identificar facilmente os recursos mais tarde, quando você pode ter muitos deles. Por exemplo, muitas pessoas usam um prefixo ou um sufixo para identificar as diferenças entre os recursos, como "obj_Ship" ou "Ship_obj".

Em seguida, você pode escolher o sprite para o object, caso deseje que ele tenha um. Clique com o botão esquerdo do mouse na caixa sprite e o Asset Explorer aparecerá com uma lista de todos os disponíveis sprites do Navegador Asset:

Você pode então selecionar aquele que deseja usar para o object. Com um sprite atribuído ao object você pode clicar no botão Edit Sprite para mudar as propriedades do sprite, ou - se você não tem um sprite para o object mas deseja criar um - você pode clicar o botão Add Sprite para criar um novo sprite asset e atribuir isso. Você também pode editar a imagem sprite asset que tem sido atribuído ao clicar no botão Edit Image para abrir o Editor de Imagem.

Máscara de colisão

Sob a área de atribuição sprite você tem um botão para escolher uma Máscara de Colisão específica. Por padrão, um object irá gerar propriedades de colisão com base no sprite atribuído a ele, mas há casos em que você pode querer colisões a serem baseadas em uma forma diferente de uma forma diferente sprite. Por exemplo, você pode ter um jogo de cima para baixo onde o principal sprite tem uma máscara de colisão retangular, mas para tornar as colisões mais suaves uma máscara circular pode ser melhor. Nesses casos você pode clicar no botão Máscara de Colisão e selecionar outro recurso sprite, então agora as instâncias do object irão desenhar o atribuído sprite, mas as colisões de base fora da máscara sprite. Você pode encontrar mais informações sobre colisões máscaras da página no Editor Sprite.

Imóveis

Aqui são apresentadas a você as seguintes opções para seu objeto:

  • Visível: Visível indica se as instâncias deste object são visíveis quando o room começa. Normalmente a maioria das instâncias são visíveis, mas às vezes é útil ter as invisíveis - por exemplo, você pode usá-las como pontos de passagem para controlar um monstro em movimento, ou para rastrear valores e executar certas ações cronometradas. Invisíveis objects ainda reagirão aos eventos e se tiverem um sprite ou uma máscara atribuída a eles, também reagirão se outras instâncias colidirem com eles... eles simplesmente não podem ser vistos e não realizam seu Evento de Sorteio. Por padrão, o visível flag é verificado.
  • Sólido: Quando você flag uma instância como sendo sólida, você está dizendo ao GameMaker que ele deve tentar resolver qualquer colisão antes de acionar um Evento de Colisão. Esta é uma funcionalidade muito básica, e na realidade tudo o que ela faz é mover a instância de volta para a posição em que se encontrava no momento anterior à detecção de uma colisão antes de executar qualquer código ou GML Visual que esteja no próprio Evento de Colisão, e como tal é de uso limitado.
  • Persistente: Um persistente object é aquele que não desaparece quando o room é alterado, mas sim "persiste" e é transportado para o novo room. Só desaparecerá quando você o destruir explicitamente, seja com uma ação de destruição ou em código. Isto significa que se você colocar um persistente object em um room, ele ainda estará disponível em todos os rooms subsequentes e continuará a executar quaisquer ações atribuídas aos eventos em suas propriedades object, exceto o Evento Criar, que só é disparado uma vez quando é criado inicialmente e não quando você muda para um novo room. Isto é útil para quando você tem, por exemplo, um personagem principal que se move de room para room e deseja manter as variáveis dentro da mesma instância, mas você deve tomar muito cuidado para criar e destruir corretamente o persistente objects, pois você pode facilmente introduzir erros em seu projeto por engano.

Persistência e Room Camadas

Quando você cria uma instância com persistência habilitada em room, lhe será atribuída uma camada ou uma profundidade (dependendo de qual função você usou para criar a instância do object). Quando o room for alterado, e se o seguinte room não tiver uma camada com o mesmo nome ou profundidade que o atribuído, então uma nova camada será criada para a instância que está persistindo nas salas. Se você deu uma camada nomeada quando a instância foi criada, então a nova camada também terá o mesmo nome que a camada original, enquanto que se você atribuiu uma profundidade à instância, então a nova camada será nomeada "_layer_XXX", onde "XXX" é um valor hexadecimal usado para dar à camada um nome único. Estas camadas serão removidas do site room no Room End.

Outra coisa a se notar sobre camadas e instâncias persistentes é que se você tiver atribuído à instância uma camada nomeada na criação, e houver outra camada no seguinte rooms com o mesmo nome, então a instância persistente será atribuída à camada com o mesmo nome, independentemente da profundidade da camada.

Finalmente, se uma instância persistente se move para uma room com uma camada na mesma profundidade em que a instância foi criada, ela não será atribuída a esta camada, mas, em vez disso, uma nova camada será criada na mesma profundidade (seguindo a convenção de nomenclatura explicada acima).

Persistência e eventos

Como mencionado anteriormente, um persistente object só executará seu Create Event uma vez, mas também terá seus eventos Início, Fim do Jogo, Início e Fim da Sala acionados nos momentos apropriados. Entretanto, se você reiniciar o jogo (com, por exemplo, a função game_restart() ), todos os eventos persistentes objects serão removidos e só existirão quando criados novamente pelo jogo.

Observe também que se você desativar um persistente object, ele não passará mais de um room para outro a menos que seja reativado antes que o evento Room End seja acionado. A reativação requer uma etapa completa do jogo para que ocorra, portanto, usar o evento Room End real para isto não funciona, pois não dá ao jogo uma etapa completa para reativar o object, então você precisa ter isto em mente ao usar a desativação junto com o persistente objects e planejar seu código de acordo.

  • Utiliza a Física: Ao verificar a caixa Física de usos, você está dizendo ao GameMaker que este object deve fazer parte de um mundo da física (como definido noEditor de Sala ). Ao verificar isto, você também abrirá uma outra janela acorrentada onde você poderá definir as propriedades físicas de todas as instâncias deste object. Mais informações sobre isto podem ser encontradas na seção sobre Objetos Físicos.
  • Gerenciado: Se este object deve ser gerenciado ao utilizar o Rollback Multiplayer. O estado de um objeto gerenciado pode ser revertido e as previsões podem ser feitas. Veja Managed Objects (Objetos gerenciados).
